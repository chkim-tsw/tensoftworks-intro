1 사업 이해도

프로젝트 본질

겉으로는 하나의 쇼핑몰이지만, 안에는 세 가지 완전히 다른 비즈니스 모델이 돌아갑니다.

유통 (원자재)

본사가 직접 팔고, 직접 배송하고, 직접 CS. 정산도 없는 자사몰 구조. 향후 입점 기능을 열 수 있게 설계만 열어둠.

판촉 (판촉물)

입점사가 상품 올리고, 주문 받고, 배송하고, CS 1차 대응. 본사는 수수료 떼고 정산해주는 마켓플레이스 구조.

실사출력

판촉과 동일한 입점 구조 + 주문제작 방식. 사이즈 직접 입력, 디자인 파일 업로드, 실시간 견적 계산 추가.

이 프로젝트의 난이도는 결국 "하나의 결제에서 시작된 주문이 판매자별로 갈라져서 각각의 배송 → CS → 구매확정 → 정산 → 세금계산서 → ERP까지 흘러가는 전체 파이프라인"을 얼마나 깔끔하게 설계하느냐에 달려 있습니다.

BO와 SCM은 동일한 기능을 공유하는 부분이 있으나, 접근 정책이 완전히 다릅니다.
BO는 정산·ERP 연동 등 민감한 데이터를 다루기 때문에 통제된 인프라(사내망/VPN) 안에서만 접속 가능하도록 설계합니다. SCM은 입점 셀러가 외부에서 접속해야 하므로 공개 네트워크에 노출됩니다. 이에 따라 인증·보안 수준도 차등 적용합니다.

설계 영향: BO와 SCM은 API 백엔드를 공유하되, BO 전용 API(정산 상세, ERP 연동, 마진율 설정 등)는 내부 네트워크에서만 접근 가능한 별도 엔드포인트로 분리합니다. SCM API는 외부 노출이므로 Rate Limiting, IP 화이트리스트(선택), 강화된 인증 토큰 정책을 적용합니다.
KT alpha 기프티쇼비즈 판촉몰에서 FO/BO/SCM/SRM 4개 시스템을 풀스크래치로 구축한 경험이 있습니다. 초기 기획 시 BO 메뉴가 40개였으나, 운영 과정에서 브랜드관리, 메인배너관리, MDPICK관리, CS관리 세분화 등이 추가되어 최종 60개 이상으로 확장되었습니다. 이 경험을 바탕으로 애드피아몰도 초기 설계부터 확장을 전제로 메뉴 구조를 잡겠습니다.

2 기능별 설계 방향

2.1 상품등록 체계

판촉물 특성상 옵션 조합이 곱하기로 늘어나는 구조입니다. 인쇄 방식, 소재, 수량 구간별 단가, 사이즈, 색상 — 이게 다 조합이니까 가격 테이블이 매트릭스가 됩니다. KT alpha 판촉몰에서 이미 구현하고 셀러들이 실제로 사용하면서 안정화된 방식을 기반으로 설계합니다.

① 기본정보 탭

SCM 상품등록 > 기본정보 화면 구성
카테고리필수 · 다중 추가 가능 · 불일치 시 등록 반려 상품명필수 · 텍스트 입력 법적 허가사항선택 · 해당 시 입력 재질필수 · 텍스트 입력 규격필수 · 가로×세로×높이(mm) 썸네일 이미지500×500px 권장 · 최대 3개 · jpg/png
카테고리 다중 추가는 "판촉물이 여러 카테고리에 걸치는 경우"를 위한 것입니다. 예: 머그컵이 '사무용품'이면서 '기념품'일 수 있습니다. 다만 카테고리 불일치로 인한 반려가 초기 셀러 이탈의 주요 원인이었습니다. 카테고리 선택 시 추천 카테고리를 자동 제안하는 UI를 추가하겠습니다.

④ 칼선(디자인) 및 인쇄영역 등록 — 편집기 API 종속 영역

고객이 FO에서 시안을 직접 올려볼 수 있도록 칼선 이미지를 등록하고, 인쇄 가능 영역을 설정하는 화면입니다.

이 영역은 연동하는 편집기(에디터 API)에 완전히 종속됩니다. 칼선 이미지 형식(jpg/png/svg), 인쇄영역 좌표 포맷, 제작용 파일 규격(AI/PDF/EPS), 시안 합성 방식 등은 모두 편집기 벤더가 정의하는 스펙에 따릅니다. 착수 전 편집기 벤더로부터 아래 항목을 반드시 공유받아야 합니다.
편집기 벤더 확인 필요 사항 (FR-SHP-007)
확인 항목설명
지원 파일 형식칼선 이미지 포맷(jpg/png/svg), 해상도·용량 제한, 제작용 파일 포맷(AI/PDF/EPS)
인쇄영역 설정 방식좌표 전달 포맷(JSON/XML), 편집기 자체 팝업인지 우리 UI에서 좌표만 넘기는지
시안 합성 프리뷰편집기가 제공하는 iframe/SDK인지, 우리가 Canvas로 합성해야 하는지
API 연동 규격REST/WebSocket, 인증 방식, 콜백 URL, 상품-칼선 매핑 키
고객 업로드 파일 처리파일 저장 위치(편집기 서버/우리 서버), 용량 제한, 지원 형식

KT alpha에서는 편집기 연동 규격을 프로젝트 중반에야 받아서 SCM 칼선 등록 화면을 두 번 재작업했습니다. 이번에는 기획/설계 단계에서 편집기 벤더 미팅을 1순위로 잡고, 연동 규격 확정 후 칼선 등록 화면을 설계합니다.

칼선 등록은 셀러들이 가장 어려워하는 기능입니다. KT alpha에서 칼선 미등록 상태로 승인 요청하는 경우가 전체의 약 30%였습니다. 칼선 미등록 시 승인 요청 버튼을 비활성화하고, 등록 가이드 툴팁을 상시 노출하는 UX를 적용하겠습니다.

⑤ 상세페이지 에디터

상품 상세페이지는 WYSIWYG 에디터를 통해 텍스트 작성, 표 삽입, 이미지 첨부가 가능합니다. 이미지는 드래그앤드롭 업로드를 지원하고, 최대 해상도 제한(2000×2000px)과 용량 제한(5MB/장)을 적용합니다.

2.2 상품 승인 워크플로우

임시저장→ 승인요청→ 승인대기→ 1차승인(BO)→ 최종승인

SCM 좌측 메뉴(LNB)가 임시저장, 승인대기, 거절, 보류로 나뉘어 있어 셀러가 상태별로 상품을 관리합니다.

상태셀러 가능 행동비고

승인대기조회만 가능 (수정 불가)관리자 심사 중
거절사유 확인만 가능판매 불가 판정 · 재등록 필요
보류사유 확인 → 수정 → 재심사 요청수정 후 재승인 가능
KT alpha 초기에는 "거절"과 "보류"를 구분하지 않았습니다. 셀러가 "사소한 수정이면 바로 고쳐서 재심사 받고 싶다"는 요구가 많아 보류(재심사 가능) / 거절(완전 반려)로 분리했습니다. 이 구분이 없으면 셀러가 상품을 처음부터 다시 등록해야 해서 이탈률이 높아집니다.

대용량 엑셀 등록

엑셀 양식 다운로드 → 일괄 입력 → 업로드 → 검증 결과 리스트(성공/실패 행 표시) → 성공 건 자동 임시저장. 이미지는 ZIP으로 별도 업로드하고 파일명으로 상품과 매칭합니다.

2.3 관리자(BO) 상품 관리

상품 심사 화면

BO에서 승인대기 상품을 조회하면, 셀러가 입력한 기본정보/공급정보/옵션/칼선/상세페이지를 모두 확인할 수 있습니다. 심사 결과는 1차승인 / 보류(사유 입력) / 거절(사유 입력) 중 선택합니다.

최종승인 후에는 마진율 수정이 불가능하게 잠급니다. 이미 고객에게 노출된 가격이 변경되면 진행 중인 견적/주문에 영향을 주기 때문입니다. 가격 변경이 필요하면 상품을 비활성화 → 신규 등록하는 프로세스를 적용합니다.

2.8 견적서 → 주문 전환 / 후불·여신한도

견적서+후불이 어려운 진짜 이유: 후불 정산 로직이 일반 카드 결제와 완전히 다릅니다. 일반 카드 결제는 PG 승인 → 구매확정 → 정산이지만, 후불은 주문 발생 → 월말 집계 → 청구 → 입금 확인 → 정산입니다. 결제 시점, 매출 인식 시점, 셀러 정산 시점이 전부 다르기 때문에, 사실상 "일반 쇼핑몰"과 "후불제 몰"을 동시에 운영하는 구조가 됩니다. 이 이중 구조는 안정화까지 상당한 시간이 소요됩니다.

견적서 발급 및 주문 전환

견적서 발급→ 고객 승인→ 주문서 자동 생성→ 결제 (또는 후불 배정)→ 입점사 전달
  • 견적 유효기간 설정 가능 (기본 7일, 관리자 조정)
  • B2B 대량 주문: 엑셀 업로드 → 견적 일괄 생성
  • 견적 승인 → 주문서 자동 생성 시 상품 옵션/수량/금액 자동 복사
  • 후불결제 계약: 법인 날인 기반 계약서 이메일 발송 / PDF 다운로드·인쇄

취소/환불은 단순한 CS 처리가 아니라 셀러 정산과 직접 엮이는 회계 이슈입니다. 장바구니에 담긴 상태에서의 취소, 결제 후 부분 취소, 배송 후 반품 — 각각이 셀러 정산에 미치는 영향이 다릅니다.

KT alpha 초기에는 "주문취소요청목록/반품완료목록/교환완료목록"으로 상태 중심으로 메뉴를 나눴다가, 실제 운영에서 "내가 처리해야 할 취소 건은 어디 있지?"라는 혼란이 발생했습니다. "취소관리/반품관리/교환관리"로 처리 유형 중심으로 변경한 후 운영 효율이 크게 개선되었습니다. 처음부터 유형 중심으로 메뉴를 잡겠습니다.

2.11 전시 / 기획전 / 이벤트 관리 — FO 개발의 핵심

전시/기획전/이벤트는 사용자 FO 개발에서 가장 중요하고 가장 위험한 영역입니다. 입점몰에서 상품 데이터는 셀러들이 채워주므로 공유 자산입니다. 결국 FO의 차별화는 "상품을 어떻게 보여주느냐" — 즉 전시/이벤트 운영에 달려 있습니다. 이것은 사실상 MD(머천다이저)의 기능 전부이며, 일반 쇼핑몰의 핵심입니다.
문제는 이 영역의 요구사항이 끝없이 확장될 수 있다는 것입니다. "타임세일", "쿠폰 자동 적용 기획전", "구매 조건부 사은품", "n+1 증정", "카테고리별 할인율 차등" 등 기획전/이벤트의 변형은 무한합니다. 1차 오픈 범위를 명확히 픽스하지 않으면, 이 영역이 오픈 지연의 직접적인 사유가 됩니다.

1차 오픈 범위 (권장)

전시관리 — Phase 1 (오픈 필수)
├ 카테고리 관리 — 추가/수정/삭제/노출 순서 변경
├ 메인배너 관리 — PC/모바일 이미지 각각 등록 · 시작/종료일 · 링크 URL
├ 띠 배너 관리 — 상단/중간/하단 위치별 등록
├ 베스트100 — 판매량/조회수 기반 자동 집계 + 수동 조정
├ MD PICK — 관리자 수동 큐레이션 상품 선택
└ 브랜드관 — 브랜드 등록 (백색/컬러 로고, 최대 2MB) · 브랜드별 상품 연결
이벤트/기획전 — Phase 2 또는 범위 협의 필수
├ 할인 이벤트 — 기간/카테고리/상품별 할인율 설정
├ 증정 이벤트 — 구매 조건부 사은품 자동 배정
├ 타임세일 — 시간 제한 할인, 재고 카운트다운
├ 쿠폰 연동 기획전 — 기획전 진입 시 쿠폰 자동 발급
└ 복합 이벤트 — n+1, 세트 할인, 단계별 할인 등
1차 오픈 시 전시관리(배너/카테고리/큐레이션)는 필수로 포함하되, 이벤트/기획전의 세부 유형은 발주사와 범위를 명확히 협의한 후 확정합니다. 범위 미확정 상태로 개발에 돌입하면 "이것도 넣어달라, 저것도 넣어달라"가 반복되면서 FO 오픈이 지연됩니다. 2차 개발로 분리하거나, 1차에 포함할 이벤트 유형을 3가지 이내로 픽스하는 것을 강력히 권장합니다.
KT alpha에서도 브랜드관리, 메인배너관리, MDPICK관리는 초기 기획에 없었으나 운영 2개월 차에 필수로 추가되었습니다. 이벤트/기획전은 오픈 후에도 지속적으로 유형이 추가되는 영역이므로, 확장 가능한 이벤트 엔진 구조(이벤트 유형 → 조건 → 혜택을 조합하는 규칙 엔진)를 설계해두면 2차 개발 시 빠르게 대응할 수 있습니다.

3 의견이 필요한 영역

3.1 사방넷 연동 — 구조적 미스매치

구분사방넷의 설계 철학애드피아몰이 필요한 구조
관계 구조1 셀러 : N 쇼핑몰
하나의 판매자가 여러 채널 주문을 한 곳에서 관리		N 셀러 : 1 쇼핑몰
여러 입점사가 하나의 몰에 상품을 올리는 마켓플레이스
재고 관리SKU 기반 재고 추적이 핵심판촉물/실사출력은 주문제작(make-to-order)이 대부분으로 SKU 재고 추적이 무의미
사방넷 연동은 Phase 2로 분리하고, Phase 1에서는 자체 재고관리(유통 카테고리) + 주문제작 프로세스 관리(판촉/실사출력)로 구조를 잡는 것이 현실적입니다.

3.2 다국어(한/영/일) — 결제까지 포함인지 확인 필요

구분시나리오 A: 국내 외국인 UI 지원시나리오 B: 해외 거주 외국인 결제
결제국내 PG 일반결제 + 간편결제 그대로해외특약 별도 계약, 3DS 인증 해외카드
간편결제네이버페이, 카카오페이 사용 가능사실상 불가 (한국 휴대폰 필수)
수수료국내 일반 수수료해외카드 3.8~4% 수준
난이도UI만 i18n → 낮음PG 해외특약 + 일본어 결제창 별도 확인 → 높음
시나리오 A는 기본 범위, 시나리오 B는 해외특약 조건 확인 후 추가 범위로 분리합니다.

3.3 셀러센터 다국어

FO(고객용) 다국어는 당연합니다. 그런데 셀러센터 다국어는 다른 문제입니다. 옵션 설정, 칼선 등록, 가격 매트릭스만으로도 충분히 복잡한 SCM에 다국어 입력 필드까지 붙이면 등록 이탈률이 높아집니다.

FO만 다국어 처리, SCM/BO는 한국어. 상품 데이터 구조는 다국어 필드를 미리 설계해두되, 입력 화면은 한국어로 시작하고 번역은 관리자 측에서 처리합니다.

6 왜 우리인가 — 경험 기반의 구체적 근거
단순히 "경험이 많습니다"가 아니라, KT alpha 기프티쇼비즈 판촉몰을 직접 구축·운영한 경험에서 나온 구체적 설계를 적용합니다.
근거구체적 내용
동일 도메인
풀스크래치 구축		판촉물/실사출력 입점몰 — FO/BO/SCM/SRM 4개 시스템을 처음부터 끝까지 직접 설계·개발.
상품등록(옵션 매트릭스, 칼선, 시안), 주문제작 7단계 프로세스, 이중 정산(판매정산+공급정산), 셀러 가입심사·서브회원·권한체계까지 전체 범위 구현.
운영에서
검증된 설계		코드정의서 기준 17건 이상의 기획-구현 간 차이 조율:
• "승인대기" 대신 "임시저장" 도입 → 셀러 이탈률 감소
• CS 메뉴를 상태 중심 → 처리 유형 중심으로 구조 변경
• 출금정산관리를 판매정산/공급정산으로 분리 → 운영 명확성 향상
• 브랜드관리/메인배너/MDPICK 등 운영 필수 메뉴 추가
• FAQ에 '시안' 카테고리 분리 (전체 문의의 40% 차지)
화면 단위
디테일 설계 역량		SCM 매뉴얼 기준 25개 이상의 화면, BO 매뉴얼 기준 60개 이상의 메뉴를 직접 기획·개발. 각 화면의 필드 구성, 유효성 검증 규칙, 상태별 UI 동작(버튼 활성화/비활성화, 팝업 조건 등)까지 실운영에서 검증된 설계 패턴을 보유.
보안 대응
ISMS-P		DB 암호화(평문 저장 지적 후 수정), API 보안 검토, 소스코드 품질점검, 취약점 진단 대응 경험. 설계 단계부터 보안 요구사항 반영.
